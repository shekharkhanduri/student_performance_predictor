# Project Report

## 1. Title & Abstract

### Project Name
Student Performance Predictor

### Abstract
The Student Performance Predictor is an end-to-end machine learning system designed to forecast student academic performance scores. The prediction logic relies on a robust Stacking Ensemble Regressor that integrates three powerful non-linear base learners: XGBoost, CatBoost, and Random Forest. These are unified by a LassoCV meta-learner to maximize predictive accuracy. The system consumes data from two distinct sources: Dataset 1, a 10,000-sample dataset with 5 core features, and Dataset 2, containing over 6,600 samples with 19 detailed features. To ensure interpretability alongside accuracy, the pipeline asynchronously computes per-student SHAP (SHapley Additive exPlanations) diagnostics in the background, offering transparent insights into the primary factors influencing each score. The deployment strategy features a FastAPI backend providing a robust, versioned REST API, paired with a modern React and Vite frontend. This full-stack application allows institutions to seamlessly upload bulk datasets via CSV, trigger background analysis, and intuitively monitor real-time tracking metrics through an active data dashboard.

## 3. Modular Structure

*   **Training_model/student_performance_predictor.py:** Serves as the standalone machine learning training pipeline for the project. It handles data preprocessing, base model training, ensemble creation, and exporting trained models to joblib files.
*   **backend/main.py:** Bootstraps the FastAPI application by initiating database migrations, configuring CORS, and prewarming machine learning models into memory. It connects the API routers to the main server instance to handle incoming requests.
*   **backend/run.py:** Provides the necessary entry point to launch the FastAPI server. It simply configures and executes the Uvicorn server on the designated host and port.
*   **backend/core/config.py:** Acts as the central configuration hub for environment variables and project constants. It defines database URLs, upload limits, and standardized API prefix rules.
*   **backend/core/database.py:** Establishes the SQLAlchemy engine and SessionLocal factories. It manages the fundamental connection required to communicate with the PostgreSQL backend.
*   **backend/core/migrations.py:** Handles database schema creation and idempotent DDL tasks dynamically. It ensures that required tables and necessary indexes are properly formed without conflicts.
*   **backend/models/student.py:** Contains the SQLAlchemy ORM models corresponding to the underlying datasets. It natively tracks student identifier columns and fields like shap_status for monitoring background tasks.
*   **backend/schemas/prediction.py:** Implements strict Pydantic models for data validation and API payloads. It standardizes the format for incoming prediction requests and outgoing UI-ready diagnostics.
*   **backend/services/ml_service.py:** Orchestrates the machine learning inference logic, spanning both initial fast predictions and slower asynchronous SHAP computations. It relies on the prewarmed ensemble models to dynamically generate accurate prediction results and diagnostic explanations.
*   **backend/services/student_service.py:** Provides pure business logic mapping to interface with the database effectively. It encapsulates CRUD operations for saving student details, updating their records, and fetching comprehensive background statuses.
*   **backend/api/deps.py:** Contains common FastAPI dependency injection functions used across different route handlers. It is responsible for consistently yielding active database sessions, validated models, and necessary scalers.
*   **backend/api/routers/health.py:** Supplies a dedicated health endpoint to check the pulse of the web server. It provides immediate status information for the API, database connectivity, and the state of preloaded machine learning models.
*   **backend/api/routers/students.py:** Functions as the core routing API for student data handling, encompassing bulk CSV uploads and specific diagnostic polling. It orchestrates HTTP 202 acceptance limits and delegates heavy calculation logic to background tasks.

## 4. Methodology

*   **Data Acquisition:** Structured tabular data is ingested via two pathways: historical standalone CSV files fed straight into the Training_model script, or newly observed instances delivered real-time via bulk CSV processing endpoints mapped to backend/api/routers/students.py. Legacy inputs featuring student IDs map automatically to an external_id for deduplication.
*   **Preprocessing:** Numerical features undergo variance scaling via StandardScaler, while distinct categorical fields receive specific label-encoding strategies. Mean or mode imputation algorithms are reliably applied across incoming dataframes to nullify missing observations prior to model inference.
*   **Model Training:** A robust StackingRegressor handles core logic, employing XGBoost, CatBoost, and Random Forest estimators as base predictors to navigate non-linear relationships. Five-fold cross-validation is natively implemented, subsequently passing intermediate outputs to a generalized LassoCV meta-learner avoiding inherent overfitting risks.
*   **Result Visualization:** Outputs are classified into actionable risk buckets (Stable >= 70, Borderline 60-69, At-Risk < 60) directly consumed by the Vite/React frontend pipeline. To guarantee transparent diagnostics, asynchronous execution invokes a global SHAP KernelExplainer, calculating exact feature influences parsed immediately into comprehensive factor badges and dynamic web scorecards.

## 5. Deployment Specs

*   **Docker Configuration:** While an explicit Dockerfile is currently uncommitted in the root directory, traditional deployment containerizes the backend FastAPI application and the compiled Node.js static bundle from frontend/dist/ within a standard python:3.10-slim envelope. It binds vital libgomp1 utilities necessary for optimized machine learning calculations.
*   **CI/CD Steps:** No standalone Jenkinsfile or GitHub Actions manifest actively runs in the repository. Standard deployment pipelines generally dictate establishing a robust sequence to execute unit tests (pytest), construct the React UI artifacts (npm run build), build containerized distributions via docker build, and seamlessly deploy validated stages into environments.
