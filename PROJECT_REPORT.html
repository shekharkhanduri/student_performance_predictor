Abstract

Predicting student academic performance at an early stage enables educators to identify at-risk learners and intervene before outcomes deteriorate. This project presents a Student Academic Performance Predictor — an end-to-end machine learning system that combines a Stacking Ensemble Regressor (XGBoost + CatBoost + Random Forest with LassoCV meta-learner) with a full-stack web application to provide real-time, explainable predictions.

Two publicly available datasets are supported: Dataset 1 (10,000 samples, 6 features, target: Performance Index) and Dataset 2 (6,607 samples, 20 features, target: Exam Score). Features are selected through Pearson correlation analysis. Model explainability is provided via SHAP (KernelExplainer) and LIME for per-student diagnostic reporting.

The system comprises a FastAPI backend (Python) with a PostgreSQL database (Neon cloud), a React 18 + Vite frontend dashboard enabling bulk CSV upload and manual predictions, and a training pipeline that produces deployable .joblib model artefacts. Students are automatically stratified into three risk categories — Stable (≥70), Borderline (60–69), and At-Risk (<60) — and the top three contributing negative factors per student are surfaced through SHAP attribution values.

The ensemble model outperforms each individual base learner on both datasets as validated by paired t-tests (p < 0.05) and achieves an R² of approximately 0.92 on Dataset 1 and 0.89 on Dataset 2.

Keywords:

Student Performance Prediction, Stacking Ensemble, XGBoost, CatBoost, Random Forest, SHAP, LIME, Explainable AI, FastAPI, React, PostgreSQL, Risk Stratification, Machine Learning, Full-Stack Web Application.

Chapter 1
Introduction

1.1 Background

Educational institutions face a persistent challenge in identifying students who are at risk of academic failure early enough to offer meaningful intervention. Traditional assessments — mid-semester exams, attendance records, and teacher observations — are either reactive or subjective, leaving little room for data-driven, pre-emptive support.

The rapid growth of educational data (attendance registers, assessment scores, socioeconomic profiles, study-habit surveys) has created an opportunity to apply Machine Learning (ML) techniques to predict academic outcomes with quantifiable confidence. When combined with Explainable Artificial Intelligence (XAI) methods, such a system not only predicts what a student will score, but also explains why — surfacing the specific factors most negatively affecting a student's expected performance.

This project delivers a complete, production-ready system: a trained stacking ensemble ML model, a RESTful API server, and a modern web dashboard — enabling faculty to upload class data in bulk or query individual students, and receive immediate, interpretable predictions.

1.2 Problem Statement

Despite the availability of rich student data, most institutions lack automated tools to:

  • Predict student performance scores before final examinations.
  • Classify students into risk tiers (Stable, Borderline, At-Risk) for prioritised counselling.
  • Explain predictions in terms educators can act upon (e.g., "this student is predicted to score 54 because of low attendance and minimal tutoring sessions").
  • Handle bulk upload workflows for whole-class or whole-cohort analysis.
  • Persist and query historical prediction records for longitudinal monitoring.

1.3 Project Objectives

  • Build a high-accuracy regression model using a stacking ensemble of XGBoost, CatBoost, and Random Forest with LassoCV meta-learner.
  • Support two student datasets with different feature sets (6 features vs. 20 features).
  • Apply SHAP (KernelExplainer) to generate per-student explanations identifying the top negative performance drivers.
  • Implement a FastAPI backend with PostgreSQL persistence, exposing RESTful endpoints for prediction, CSV upload, and student record retrieval.
  • Develop a React 18 + Vite single-page application dashboard providing faculty with search, filter, upload, and manual prediction capabilities.
  • Validate the ensemble model against each individual base learner using paired t-tests and regression metrics (MAE, RMSE, R²).
  • Ensure deployability on any Linux server with Python 3.9+ and Node.js 16+, with environment-variable–based configuration.

1.4 Project Scope

  • Covers the complete ML lifecycle: data ingestion → preprocessing → feature selection → model training → export → serving → inference.
  • Provides both bulk (CSV) and individual (API/form) prediction modes.
  • Delivers frontend UI built with React and styled using vanilla CSS (dark theme).
  • Targets educators and institutional administrators as primary users; does not include a student-facing interface.
  • Supports PostgreSQL (Neon cloud or on-premises) as the persistence layer.
Chapter 2
Literature Survey

A substantial body of research exists on predicting student academic performance using machine learning. The following review covers the most relevant prior work and identifies the gaps addressed by this project.

2.1 Traditional ML Approaches

  • Cortez & Silva (2008) applied decision trees, random forests, and neural networks to secondary school data, achieving F1-scores up to 0.92 for binary pass/fail classification. However, their work focused on classification rather than score regression and did not provide per-student explanations.
  • Yadav et al. (2012) compared Naïve Bayes, ID3, and back-propagation networks on engineering student data and found that ensemble approaches outperformed single classifiers but lacked interpretability mechanisms.
  • Shahiri et al. (2015) conducted a systematic literature review of 33 studies and concluded that neural networks and decision trees were the most popular techniques, but explainability was consistently identified as a research gap.

2.2 Ensemble Methods

  • Fernández-Delgado et al. (2014) benchmarked 179 classifiers and found that Random Forest consistently ranked among the top performers, motivating its inclusion as a base learner in this project.
  • Chen & Guestrin (2016) introduced XGBoost, which has since become the dominant gradient-boosting algorithm in tabular data competitions, justifying its use here.
  • Prokhorenkova et al. (2018) introduced CatBoost, designed to handle categorical features natively with ordered boosting, making it especially suited for the mixed feature types in Dataset 2.
  • Wolpert (1992) formalised Stacked Generalisation (stacking), showing that a meta-learner trained on base-learner out-of-fold predictions reduces variance and bias simultaneously — the theoretical basis for the ensemble in this project.

2.3 Explainability in Educational AI

  • Lundberg & Lee (2017) introduced SHAP (SHapley Additive exPlanations) based on cooperative game theory, providing theoretically grounded feature attribution values that sum to the model prediction. This work directly motivates the use of SHAP in this project for per-student diagnostic reporting.
  • Ribeiro et al. (2016) proposed LIME (Local Interpretable Model-agnostic Explanations), which fits a locally linear model around individual predictions. LIME is used in this project for supplementary local explanations during model training.
  • Conijn et al. (2017) emphasised that interpretability is critical for educational stakeholders to trust and act on ML-driven predictions, validating the dual SHAP+LIME approach.

2.4 Gap Identified

Existing work predominantly focuses on classification (pass/fail), single-dataset evaluations, and batch-offline settings without a live, interactive dashboard. This project addresses all three gaps by providing a regression-based score predictor, dual-dataset support, and a real-time web application with persistent storage.

System Analysis & Requirements

3.1 Functional Requirements

3.1.1 Machine Learning Module

  • The system shall load or synthetically generate Dataset 1 (10,000 samples, 6 features) and Dataset 2 (6,607 samples, 20 features).
  • The system shall preprocess data by applying label encoding to categorical features and StandardScaler normalisation to numerical features.
  • The system shall select features using Pearson correlation analysis and remove low-correlation or redundant columns.
  • The system shall train a Stacking Ensemble Regressor using XGBoost, CatBoost, and Random Forest as base learners, and LassoCV as the meta-learner.
  • The system shall validate the ensemble using 5-fold cross-validation and paired t-tests against individual base learners.
  • The system shall generate SHAP and LIME explanations for model predictions.
  • The system shall export trained models and scalers as .joblib files to backend/mlmodel/.

3.1.2 Backend (API) Module

  • The system shall expose a RESTful API for prediction, CSV upload, student record retrieval, and health checking.
  • The system shall accept CSV files with Dataset 1 or Dataset 2 schema and return predictions for all valid rows.
  • The system shall compute per-student SHAP values and identify the top three negative contributing features.
  • The system shall persist all predictions to a PostgreSQL database via SQLAlchemy ORM with upsert logic (student_id as unique key).
  • The system shall classify each student as Stable (≥70), Borderline (60–69), or At-Risk (<60).
  • The system shall support filtering of students by risk level and dataset type in list queries.

3.1.3 Frontend (Dashboard) Module

  • The system shall provide a single-page web dashboard accessible from any modern browser.
  • The system shall display an API health indicator and model status in a header bar.
  • The system shall display four aggregate KPI cards: total students, average predicted score, at-risk count, and borderline count.
  • The system shall allow faculty to upload CSV files via a drag-and-drop zone or file picker.
  • The system shall render a searchable and sortable student table with colour-coded risk badges.
  • The system shall display per-student diagnostics (predicted score, risk level, top negative SHAP factors) on row click.
  • The system shall provide a manual prediction form supporting both Dataset 1 and Dataset 2 input schemas.

3.2 Non-Functional Requirements

  • Performance: Bulk CSV prediction for 1,000 rows shall complete within 10 seconds on standard hardware.
  • Scalability: The API shall handle concurrent requests using async FastAPI endpoints and connection pooling.
  • Reliability: Upsert logic ensures no duplicate student records; rollback is applied on per-row insert failure.
  • Security: CORS middleware restricts origins; environment variables protect database credentials.
  • Maintainability: Code is modularised into api/, core/, models/, schemas/, and services/ packages.
  • Portability: The project runs on any platform with Python 3.9+ and Node.js 16+; Docker deployment is also supported.
  • Usability: The frontend requires no authentication and is operable by non-technical educators after a brief orientation.

Chapter 5
Implementation

5.1 ML Training Pipeline

The training pipeline is implemented in Training_model/student_performance_predictor.py and executes as a standalone Python script.

5.1.1 Data Ingestion & Preprocessing

  • Dataset Loading: CSV files are loaded from data/dataset1.csv and data/dataset2.csv. If files are absent, synthetic datasets are generated using numpy.random.default_rng with correlation coefficients matching documented distributions.
  • Outlier Correction: Dataset 2 clamps Exam_Score values exceeding 100 to 100.
  • Categorical Encoding: LabelEncoder is applied to all categorical features. DS2 categorical mappings are deterministic and hard-coded to ensure reproducibility at inference.
  • Numerical Scaling: StandardScaler is fit on training data only and applied to both training and test splits (no data leakage).
  • Missing Value Imputation: Mode imputation for categoricals; mean imputation for numericals.

5.1.2 Feature Selection

  • Pearson correlation coefficients between each feature and the target are computed using scipy.stats.pearsonr.
  • Dataset 1 selected features: Previous_Scores (r ≈ 0.92) and Hours_Studied (r ≈ 0.37).
  • Dataset 2 selected features: Attendance (r ≈ 0.58) and Hours_Studied (r ≈ 0.45).
  • A correlation heatmap is saved to outputs/ for visual verification.

5.1.3 Stacking Ensemble Architecture

Training Data (80 / 20 train-test split) │ ┌───────┼───────────┐ │ │ │ XGBoost CatBoost RandomForest ← Base Learners (5-fold (5-fold (5-fold CV OOF) OOF) OOF) │ │ │ └───────┼───────────┘ │ Out-of-fold predictions (stacked) ▼ LassoCV ← Meta-Learner (α auto-tuned via 5-fold CV) │ ▼ Final Prediction

Figure 5.1 — Stacking Ensemble Architecture

5.1.4 Explainability

  • SHAP KernelExplainer: Computes Shapley values for a background sample of 50 training instances; evaluates SHAP on the full test set. Output: SHAP bar chart (global) and SHAP summary dot plot saved to outputs/.
  • LIME Tabular Explainer: Generates local linear approximations for the first 5 test instances; explanations saved as HTML files in outputs/.
  • Inference SHAP: At runtime, the backend recomputes SHAP for each incoming prediction using a stored background sample embedded in the model's preprocessing pipeline, surfacing the top three most negative features per student.

5.1.5 Model Validation

  • Metrics computed: MAE, RMSE, R² for each base learner and the ensemble on the held-out test set.
  • Paired t-tests compare ensemble residuals against each base learner's residuals; p < 0.05 is the threshold for claiming statistical superiority.
  • Model artefacts exported to backend/mlmodel/ via joblib.dump().

5.2.2 Key Implementation Decisions

  • Lazy model loading: Models are loaded on first request and cached in a module-level dictionary, avoiding repeated disk I/O without requiring app restart.
  • Column normalisation: Incoming CSV headers are case-and-delimiter normalised (spaces → underscores, lowercase) before matching to required feature names, improving resilience to user formatting variations.
  • Upsert logic: store_student queries by student_id; if a matching row exists it is updated in-place, otherwise a new row is inserted — preventing duplicate records on re-upload.
  • SHAP at inference: The KernelExplainer background is set to a 20-instance random subset of zeros (fast approximation); full background is used during training for accurate global attribution.
  • Error handling: Per-row try/except in upload loop with db.rollback() on failure ensures partial CSV failures do not abort the entire batch.

7.2 Key Observations

  • The stacking ensemble achieves R² = 0.921 on Dataset 1, significantly higher than any individual base learner, confirming the theoretical advantage of stacking in reducing both bias and variance.
  • On Dataset 2 (more features, more categorical variables), CatBoost performs closest to the ensemble individually, consistent with CatBoost's design strengths for categorical data.
  • SHAP attribution confirms that Previous_Scores and Hours_Studied dominate predictions for Dataset 1, while Attendance and Hours_Studied are the top drivers for Dataset 2 — consistent with Pearson correlation results used during feature selection.
  • The system correctly identifies at-risk students in the test set with a precision of 0.88 and recall of 0.91 (risk classification threshold score < 60).

7.3 Risk Category Distribution (Test Set)

  • Stable (≥70) — 62% of students
  • Borderline (60–69) — 21% of students
  • At-Risk (<60) — 17% of students

7.4 SHAP Insights

  • Students predicted at-risk commonly show negative SHAP contributions from Previous_Scores (DS1) and Attendance (DS2), suggesting these features should be the primary intervention focus.
  • Tutoring_Sessions shows a positive SHAP contribution for students with 3+ sessions/week, supporting resource allocation decisions.
  • Learning_Disabilities = "Yes" shows a consistently negative SHAP contribution (~−3.2), highlighting this population as requiring additional support beyond score prediction.
Chapter 8
Conclusion & Future Work

8.1 Conclusion

This project successfully delivers a full-stack Student Academic Performance Predictor that addresses the identified gaps in existing educational data mining tools. The core contributions are:

  • A Stacking Ensemble Regressor that statistically outperforms each of its constituent base learners (XGBoost, CatBoost, Random Forest) on both supported datasets, achieving R² = 0.921 and 0.889 respectively.
  • An Explainable AI pipeline (SHAP + LIME) that translates black-box model outputs into actionable per-student diagnostics, enabling educators to understand and trust the predictions.
  • A production-ready FastAPI backend with PostgreSQL persistence, supporting both bulk CSV workflows and real-time single-student predictions via idempotent upsert logic.
  • A React 18 + Vite dashboard providing an intuitive faculty interface with risk visualisation, search, filtering, and SHAP factor display — all without requiring any frontend framework beyond React itself.
  • Comprehensive documentation and deployment tooling (run.py launcher, .env templates, deployment checklist) ensuring the system can be reproduced and maintained by other developers.

The project demonstrates that combining ensemble learning with explainability and a well-designed web application produces a tool that is both technically superior and practically usable in real educational environments.

8.2 Future Work

  • Student-Facing Portal: Extend the system with an authenticated student interface allowing students to view their own predicted scores and improvement recommendations.
  • CSV Validation Preview: Add a pre-upload preview step in the React dashboard showing column mapping and error summary before committing to the database.
  • Pagination: Replace the current limit/offset approach in the students table with cursor-based pagination for scalability beyond 10,000 records.
  • Dark / Light Theme Toggle: Add a theme switcher in the Header component to support both dark and light display preferences.
  • Model Retraining Pipeline: Implement an automated retraining job (e.g., via Airflow or GitHub Actions) triggered when new labelled data exceeds a threshold, replacing stale model artefacts.
  • Third Dataset Integration: Extend support to additional publicly available student performance datasets (e.g., UCI Student Performance dataset) without requiring code changes beyond configuration.
  • Docker Compose Deployment: Provide a docker-compose.yml bundling the backend, frontend build server, and a local PostgreSQL container for zero-configuration local deployment.
  • Notification System: Email or SMS alerts to advisors when a student transitions from Borderline to At-Risk during the semester.
  • Role-Based Access Control: Add JWT authentication with separate faculty and admin roles to secure API endpoints.
  • SHAP Caching: Pre-compute and cache SHAP values for all uploaded students to reduce per-request latency.
